# Rule-Based Language with Execution and Analysis

In this repository you will find a program that implements a rule-based language interpreter made by Jerónimo Echeverry and Sofía Marín.

## What does it do?

The system allows you to define rules of the form `if <condition> then <action>` and evaluate them against an initial state. Starting from a set of known facts and variable values, the interpreter applies all matching rules repeatedly until no new facts are produced. This process is called fixed-point evaluation.

The system is built following the same pipeline as a compiler:

* **Lexer** — reads the raw text and converts it into tokens (the smallest meaningful units of the language).
* **Parser** — takes the tokens and verifies that the structure follows the grammar, then builds an Abstract Syntax Tree (AST).
* **AST** — a tree structure that holds the program in memory so the interpreter and analyzer can use it.
* **Interpreter** — walks the AST and evaluates the rules against the current state, repeating until no new facts are activated.
* **Static Analyzer** — inspects the rules without executing them and reports conflicts, redundancies, and potentially inactive rules.

## Language

A program consists of rules followed by an optional initial state:

```
rule r1:
if temp > 30 then alert

State:
temp = 35
```

Allowed conditions are comparisons (`id > n`, `id < n`, `id = n`), active facts (`id`), and conjunctions with `AND`. Each rule can only have one action. Keywords `rule`, `if`, `then`, and `AND` are case-sensitive.

## Output

Activated facts are printed one per line in alphabetical order. If no facts are activated, the output is exactly:

```
(no output)
```

Static analysis messages are printed after the execution output:

```
Action fan_on generated by r1, r2
Redundant rules: r1, r2
Potentially inactive rule: r3
```

## How to Run

1. Clone this repository:

```
git clone https://github.com/JeroEcheverry/Rule-Based-Language-with-Execution-and-Analysis.git
```

2. Open the project in CLion (File → Open → select the project folder). Note: this project uses CMake, so it is recommended to use CLion or any IDE that supports CMake projects.

3. CLion will automatically detect the `CMakeLists.txt` and configure the project.

4. Click the green Run button at the top right (or press Shift + F10).

5. The program will start and show a menu in the terminal where you can:
   * Type or paste a program directly into the console (type `END` on a new line when finished)
   * Load a program from a `.txt` file by entering the file path

**Alternatively, using the terminal:**

Navigate to the `src/` folder and compile manually:

```
g++ main.cpp lexer.cpp parser.cpp interpreter.cpp Analyzer.cpp -o main
```

Then run with a file:

```
./main ../tests/case1.txt
```

Or run all test cases at once from the root of the project:

```
chmod +x run_tests.sh
./run_tests.sh
```

## Project Structure

* `ast.h` — defines all AST node types: `CondNode`, `AndNode`, `CmpNode`, `FactNode`, `RuleNode`, and `ProgramNode`.
* `lexer.h` and `lexer.cpp` — implements the lexer that converts raw text into a list of tokens.
* `parser.h` and `parser.cpp` — implements the LL(1) recursive descent parser that builds the AST from the token list. Includes `parseState()` to read the initial state.
* `interpreter.h` and `interpreter.cpp` — implements the fixed-point evaluation loop and recursive condition evaluation.
* `Analyzer.h` and `Analyzer.cpp` — implements static analysis for conflicts, redundancies, and potentially inactive rules.
* `main.cpp` — manages user input and connects all modules in order: lexer → parser → interpreter → analyzer → output.
* `tests/` — contains all 9 test cases from the project specification as `.txt` files.
* `run_tests.sh` — shell script that compiles the project and runs all test cases automatically.
